## 四、奖励函数设计

### 4.1 步骤奖励（每步执行后）

```
r_step = g_hop                              # 跳数惩罚（鼓励短路径）
       + α₁ × (g_node(v_j) + g_link(e_ij)) # 节点/链路剩余容量奖励
       - α₂ × c_distr(v_j)                 # 负载均衡成本惩罚
       - w_d × delay_penalty                # 链路时延惩罚（直接优化T_up）
       + r_compete × flows_on_link          # 竞争惩罚（避免多流集中）
       ± r_diversity                        # 路径多样性奖励（可选）
```

**各分量说明：**

| 符号 | 值 | 含义 |
|------|----|------|
| `g_hop` | -1.0 | 每跳固定惩罚，鼓励选短路径 |
| `α₁` | 0.4 | 负载奖励权重 |
| `g_node(v_j)` | `1 - load(v_j)/cap(v_j)` | 目标节点剩余容量比例 |
| `g_link(e_ij)` | `eff_bw(i,j) / max_cap(i,j)` | 链路有效带宽比例（考虑竞争） |
| `α₂` | 0.1 | 负载均衡惩罚权重 |
| `c_distr(v_j)` | `(2/π)·arctan(load(v_j) - avg_load(邻居))` | 节点负载分布成本 |
| `w_d` | 0.8 | 时延惩罚权重 |
| `delay_penalty` | `min((M+L)/eff_bw / 20, 1)` | 归一化链路传输时延 |
| `r_compete` | -0.5 | 每条活跃竞争流的惩罚 |

**特殊奖励：**
- 到达Server：`+r_success = +20.0`
- 路由失败（超步/无效动作）：`r_fail = -5.0`
- 环路检测（重访节点）：`r_loop = -2.0`

### 4.2 Episode 级奖励（所有GBS完成后）

```
r_episode = β_tup × exp(-2 × min(T_up/50, 1))
```

| 参数 | 值 | 说明 |
|------|----|------|
| `β_tup` | 5.0 | T_up奖励权重 |
| T_up = 5s | +4.49 | 快速完成的高奖励 |
| T_up = 30s | +1.65 | 中等完成的奖励 |
| T_up = 60s | +1.35 | 慢速完成的低奖励 |

**设计意图**：使用指数衰减而非线性归一化，拉大好策略（T_up≈5s）和次优策略（T_up≈60s）之间的奖励差距（约3.1分），引导A3C收敛到全局最优。

### 4.3 FL时延模型

```
T_path(p) = Σ_{l∈p} (M + L) / eff_bw(l)  +  hop_delay × len(p)

T_up = max_{i∈GBS} T_path(upload_i)        # 掉队者效应

T_round = T_up + T_agg
```

其中 `eff_bw(l) = base_bw(l) / (active_flows(l) + 1)` 考虑了并发竞争。

## 五、改进A3C算法的四项创新

### 创新一：竞争感知状态空间

**原版A3C状态**：仅含当前智能体自身位置和局部网络特征。

**本文改进**：设计了面向多流并发场景的**竞争感知状态向量**（62/82/90维，随GBS数变化）：

```python
obs = [
    cur_gbs_id / num_gbs,          # 当前决策GBS编号（归一化）
    cur_pos / num_nodes,            # 当前节点位置
    all_gbs_pos[0..N-1] / num_nodes,# 所有GBS当前位置（竞争感知核心）
    server_id / num_nodes,          # 目标节点
    slot_norm,                      # 当前时隙（时变感知）
    node_load_features[0..7],       # 节点负载率向量
    link_load_features[0..17],      # 链路负载率向量
    link_active_flows[0..17],       # 链路活跃流数（竞争量化）
    visited_flags[0..7],            # 已访问节点标记（防环路）
    gbs_done_flags[0..N-1],         # 各GBS完成状态
]
```

**效果**：Agent能感知其他GBS的位置，学会"让路"和负载分散，而非独立贪婪决策。

### 创新二：FL时延驱动的复合奖励函数

**原版A3C**：简单标量奖励，与FL任务目标无关。

**本文改进**：设计了五分量复合奖励，将"最小化FL单轮聚合时延T_up"这一全局稀疏目标分解为每步密集奖励：

- 步骤奖励：跳数惩罚 + 负载感知 + 竞争惩罚 + 时延惩罚
- Episode奖励：指数衰减的T_up直接奖励

**关键创新**：指数衰减奖励 `β·exp(-2T_up/50)` 拉大了好策略和次优策略之间的奖励差距，解决了多峰优化中的局部最优陷阱问题。

### 创新三：并发多流Episode建模

**原版A3C**：单智能体、静态环境。

**本文改进**：设计了**并发多流Episode**：

```
Episode = N个GBS同时出发，轮流各走一跳，直到所有GBS到达Server

每步：
1. 当前GBS执行一跳决策
2. 更新链路活跃流计数（竞争建模）
3. 轮换到下一个未完成GBS

T_up = max(所有GBS到达时间)  ← 掉队者效应
```

这比串行设计更真实地模拟了FL多客户端并发上传场景，使Agent能感知并利用多流竞争信息。

### 创新四：改进A3C训练机制（五项修复）

| 修复项 | 原版问题 | 本文方案 |
|--------|---------|---------|
| **n-step TD** | Monte Carlo回报方差大 | n=5步TD估计，降低梯度方差 |
| **梯度裁剪** | 高负载场景梯度爆炸 | max_norm=0.5，稳定训练 |
| **熵系数退火** | 固定熵系数阻止收敛 | 从0.05线性退火到0.002 |
| **更新频率** | 每episode更新1次 | 每3步更新1次，提高样本效率 |
| **Advantage归一化** | 梯度尺度不稳定 | 每批次归一化Advantage |

### 6.2 核心结论分析

**1-UAV场景**：
- A3C 与 LAD 持平（均值2.28s，中位数1.98s），优于 DQN（+34%）
- 验证改进算法不损害单流性能

**3-UAV场景（核心结论）**：
- 均值：A3C **优于 LAD 37.5%**，**优于 DQN 8.0%**
- 中位数：A3C 4.35s，略优于 LAD 4.37s，优于 DQN 5.17s
- 坏场景率：A3C **11.3%** < DQN 12.0% < LAD 20.3%（最稳健）
- A3C学会了让3个UAV走3条不同路径（Diversity=3），避免多流竞争

**5-UAV场景**：
- 均值：A3C **31.75s，优于 LAD（+19.8%）、DQN（+74.3%）、SP（+56.4%）**
- 中位数：A3C 6.77s，与 LAD（6.94s）持平，**远优于 DQN（114.74s）**
- 坏场景率：A3C **22.7%**，显著低于 DQN（92.7%）和 LAD（30.0%）
- **DQN在5-UAV场景训练失败**（median=114s，bad=92.7%），说明DQN的ε-greedy探索在高并发高维状态空间下效率极低
- A3C的竞争感知状态空间在5-UAV场景的优势更加突出

### 6.3 关键发现：A3C vs DQN 的规模效应

| 指标 | 1-UAV | 3-UAV | 5-UAV |
|------|-------|-------|-------|
| A3C均值 vs DQN均值 | A3C差34% | A3C好8% | **A3C好74%** |
| A3C坏场景率 vs DQN | 持平 | A3C好0.7pp | **A3C好70pp** |

**随UAV数量增加，A3C相对DQN的优势急剧增大**，在5-UAV场景DQN完全失效，而A3C保持良好性能。这证明了竞争感知状态空间设计的有效性。

### 6.4 LAD均值高于A3C的原因解释

LAD（负载感知Dijkstra）的T_up**高方差**：
- 3-GBS：median=4.4s，但bad rate=20.3%，均值被极端值拉高到27.4s
- 极端情况：当某条出口链路初始带宽极低（<1Mbps）时，LAD无法预判多流竞争，多个GBS都选同一条瓶颈链路，T_up极高（>100s）

A3C通过训练学到了更鲁棒的策略：**主动分散路径，避免竞争同一条链路**，因此坏场景率（9%）远低于LAD（20.3%）。

## 八、算法伪代码

### 改进A3C训练流程

```
初始化 Actor网络π_θ, Critic网络V_φ
初始化 Adam优化器（lr_actor=5e-4, lr_critic=1e-3）

For episode = 1 to N:
    重置环境，所有GBS从起点出发
    ent_coef ← 熵系数退火（0.05 → 0.002）
    
    While not done:
        cur_gbs ← 当前决策GBS
        valid_actions ← 拓扑可达邻居节点
        
        # 动作采样（动作mask保持一致）
        logits ← π_θ(state)[valid_actions]
        action ← Categorical(logits).sample()
        
        # 执行动作，获得下一状态和奖励
        next_state, reward, done ← env.step(action)
        buffer.append(state, action, reward, next_state, done, valid_actions)
        
        # 每3步或episode结束时更新
        If len(buffer) >= 3 or done:
            # n-step TD回报估计（n=5）
            returns ← n_step_TD_returns(buffer.rewards, γ=0.98)
            
            # Advantage归一化
            values ← V_φ(buffer.states)
            advantage ← (returns - values).normalize()
            
            # Critic更新
            L_critic ← MSE(values, returns)
            ∇φ ← clip(∇L_critic, max_norm=0.5)
            φ ← φ - lr_critic × ∇φ
            
            # Actor更新（带熵正则化）
            L_actor ← -E[log π_θ(a|s) × advantage] - ent_coef × H(π_θ)
            ∇θ ← clip(∇L_actor, max_norm=0.5)
            θ ← θ - lr_actor × ∇θ
            
            buffer.clear()
    
    # Episode结束：T_up奖励
    T_up ← max_i T_path(GBS_i)
    r_episode ← β_tup × exp(-2 × T_up / 50)
    最后一步奖励 += r_episode
```

## 十一、已知局限性与未来工作

### 当前局限性

1. **5-GBS场景A3C收敛不稳定**：单进程A3C在高并发场景样本效率不足，导致训练后收敛到静态路径策略，对网络状态变化响应不足。

2. **单进程训练**：真正的A3C应使用多Worker并行探索，当前因macOS上PyTorch多进程限制而使用单进程实现，失去了异步并行的核心优势。

3. **仿真系统局限**：当前为纯Python仿真，未对接AirSim物理引擎和Mininet网络仿真器，与开题报告中的跨域联合仿真系统目标尚有差距。

### 未来工作方向

1. **多进程真异步A3C**：在Linux服务器上运行真正的多Worker并行训练，解决5-GBS收敛问题。

2. **多智能体强化学习（MARL）**：使用CTDE（集中训练分布执行）框架，每个GBS作为独立智能体，共享全局状态信息，更自然地建模多流协作。

3. **图神经网络状态编码**：用GNN替代全连接层编码网络拓扑，提升对动态拓扑的泛化能力。

4. **AirSim+Mininet联合仿真**：对接物理引擎和网络仿真器，验证算法在真实协议栈下的有效性。
