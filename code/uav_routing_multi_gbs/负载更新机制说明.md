# 节点负载和链路负载更新机制详解

## 概述

本系统采用**Episode内依次更新**的负载管理机制，3个GBS在同一个Episode内依次决策，后续GBS能够看到前面GBS对网络负载的累积影响。

### 初始负载设置（第87-95行）

```python
def _init_network_state(self):
    """初始化网络状态"""
    # 节点负载：随机10-20条流表项（模拟背景流量）
    self.node_load = {i: random.randint(10, 20) for i in range(self.num_nodes)}
    
    # 链路负载：随机20%-40%的带宽使用率
    self.link_load = {}
    for edge in self.network.edges():
        self.link_load[edge] = random.uniform(0.2, 0.4)
```

## 2. 负载读取（决策时）

### 在奖励函数中读取（第109-118行）

```python
def _calculate_node_load_reward(self, node_id):
    """计算节点负载奖励"""
    load_ratio = self.node_load[node_id] / self.node_capacity
    # node_id的当前负载 ÷ 容量 = 负载率
    return 1 - min(load_ratio, 1.0)

def _calculate_link_load_reward(self, node_from, node_to):
    """计算链路负载奖励"""
    edge = (node_from, node_to)  # 有向边
    load = self.link_load.get(edge, 0.2)
    # 获取链路的当前负载率
    return 1 - min(load, 1.0)
```

**读取时机**：
- 每执行一个step，在`_calculate_reward()`中读取
- 用于计算该step的即时奖励

### 具体更新逻辑（第174-187行）

```python
def _update_network_load(self, path_nodes):
    """
    更新网络负载状态（基于完整路径）
    只有成功到达目标时才调用
    """
    # 1. 更新节点负载
    for node in path_nodes:
        self.node_load[node] = min(self.node_load[node] + 1, self.node_capacity)
    
    # 2. 更新链路负载
    for i in range(len(path_nodes) - 1):
        edge = (path_nodes[i], path_nodes[i+1])
        current_load = self.link_load.get(edge, 0.2)
        self.link_load[edge] = min(current_load + 0.08, 1.0)
```

**更新规则**：
- **节点负载**：路径上每个节点 +1 条流表项（上限50）
- **链路负载**：路径上每条链路 +8% 带宽使用率（上限100%）

## 关键设计特点

### 1. 只读不写（决策时）

```python
# 在step()过程中
step_reward = self._calculate_reward(...)  # ← 只读取负载
# 不立即更新负载

# 只有GBS成功时才写
if gbs_terminated:
    self._update_network_load(path_nodes)  # ← 写入负载
```

**为什么这样设计？**
- ✅ 保证负载的一致性（一个GBS的所有step看到相同的网络状态）
- ✅ 只有成功传输才占用资源
- ✅ 失败的尝试不影响网络

### 2. 依次累积（Episode内）

```python
初始负载 → GBS0更新 → GBS1看到变化 → GBS1更新 → GBS2看到累积变化 → GBS2更新
```

**顺序很重要**：
```
GBS0决策时: 看到初始负载
GBS1决策时: 看到初始负载 + GBS0影响
GBS2决策时: 看到初始负载 + GBS0影响 + GBS1影响
```

### 3. Episode间重置

```python
# 每个新Episode开始时
def reset(self):
    self._init_network_state()  # ← 重新随机初始化
```

**设计理由**：
- 每个Episode模拟独立的网络场景
- 初始负载随机化，增加训练样本多样性
- 避免负载无限累积

## 负载更新的数值分析

### 节点负载增长

```
容量: 50条流表项
初始: 10-20条（随机）
每次成功: +1条

最多能承受的成功传输: 
  节点5初始=19 → 最多可以+31次 → 达到50（饱和）

在1个Episode内:
  最多3次更新（3个GBS）
  节点5: 19 → 20 → 21 → 22（如果都经过）
  
结论: 1个Episode内不会饱和
```

### 链路负载增长

```
初始: 20%-40%（随机）
每次成功: +8%
上限: 100%

链路(5,7)的演变:
  初始: 35%
  GBS0经过: 35% + 8% = 43%
  GBS1经过: 43% + 8% = 51%
  GBS2避开: 保持51%（如果选择Router6）

如果3个GBS都经过同一链路:
  35% → 43% → 51% → 59%
  
结论: 1个Episode内可能接近饱和，鼓励负载均衡
```

## 负载对决策的影响

### 奖励函数中的体现

```python
reward = -1 + 0.4*(node_load_reward + link_load_reward) - 0.1*distribution_cost

示例1: 低负载路径
  node_load = 10/50 = 0.2  → node_reward = 0.8
  link_load = 0.25         → link_reward = 0.75
  reward = -1 + 0.4*(0.8+0.75) - 0.1*0.1 ≈ -0.39

  node_load = 21/50 = 0.42 → node_reward = 0.58
  link_load = 0.51         → link_reward = 0.49
  reward = -1 + 0.4*(0.58+0.49) - 0.1*0.2 ≈ -0.59

差异: -0.59 vs -0.39 = -0.20（高负载路径奖励更低）
```

**学习效果**：
- 智能体会学习到避开高负载路径
- 实现自动的负载均衡

## 总结

### 负载更新机制的核心要点

1. ✅ **初始化**：每个Episode随机初始负载（10-20节点，20-40%链路）
2. ✅ **读取**：每个step在计算奖励时读取当前负载
3. ✅ **更新**：只有GBS成功到达Server时才更新负载
4. ✅ **累积**：同一Episode内，后续GBS看到前面GBS的累积影响
5. ✅ **失败处理**：失败不更新负载，只给惩罚奖励-10
6. ✅ **重置**：每个新Episode重新初始化负载

### 这种机制的优势

- ✅ **真实性**：模拟真实的多GBS并发传输
- ✅ **一致性**：单个GBS决策过程中网络状态保持不变
- ✅ **渐进性**：负载依次累积，体现先后顺序
- ✅ **合理性**：成功才占用资源，失败不影响网络

### 负载均衡学习

通过这种机制，智能体能够学习到：
- 避开已经被前面GBS使用的高负载路径
- 在多个可行路径间智能分配
- 实现自动的负载均衡策略

这是本系统的核心创新点！
